Модели monitorRules

Модели мониторинга состояния таблиц БД позволяют организовать контроль состояния захвата, загрузки и обработки данных в таблицы БД.
 

Регистрация модели в репозитории и работа с ней

Модель мониторинга можно оформить как обычный скрипт процесса:
// Файл ModelMonitor1.groovy
package myproject.process
 
import ru.easydata.lang.EasyEtl
 
@BaseScript EasyEtl main
 
// Регистрация в репозитории соединения к Vertica и назначение его по умолчанию для запросов Vertica
useQueryConnection verticaConnection('dwh:con', true) {
  // параметры соединения
}
 
// Регистрация в репозитории модели мониторинга
monitorRules('dwh:monitor1', true) {
  // Зарегистрировать в репозитории таблицу Vertica и назначить ее в качестве таблицы хранения статусов для мониторинга
  useStatusTable verticaTable('dwh:monitor', true) {
    useConnection verticaConnection('dwh:con')
    schemaName = 'public'
    tableName = 's_monitor_tables'
  }
 
  // Количество потоков одновременно обрабатываемых правил модели
  countThreads = 2
 
  // Использовать расширение Groovy для работы с временными интервалами
  use (TimeCategory) {
    // Регистрация запроса 1 для правила в репозитории
    query('dwh:query1', true) {
      // Текст запроса
      setQuery 'SELECT Max(filedate) AS value FROM public.s_file_history WHERE filedate <= GETDATE()'
      // Правила мониторинга
      rule {
        // Проверять состояние раз в 10 минут
        checkFrequency = 10.minutes
        // Повторять уведомление про ошибку каждый час после её обнаружения
        notificationTime = 1.hour
        // Разрешать отставание загрузки файлов не более 20 минут от текущего времени сервера
        lagTime = 20.minutes
        // Описание правила
        description = 'Статус истории загрузки файлов (public.s_file_history)'
      }
    }
 
    // Регистрация запроса 2 для правила в репозитории
    query('dwh:query2', true) {
      // Текст запроса
      setQuery 'SELECT region AS code, Max(dt) AS value FROM public.s_calc_regions WHERE dt <= GETDATE() GROUP BY region'
    }
    // Описание правила на запрос
    rule('dwh:query2') {
        // Проверять состояние раз в 12 часов
        checkFrequency = 12.hours
        // Повторять уведомление про ошибку раз в сутки после её обнаружения
        notificationTime = 1.day
        // Разрешать отставание загрузки файлов не более 1.5 суток от текущего времени сервера
        lagTime = 1.day + 12.hours
        // Описание правила
        description = 'Статус истории расчета витрины по регионам (public.s_calc_regions)'
    }
  }
 
  // Вызов проверки состояния правил монитора
  if (!check())
    // Разослать письма, если в результате проверки выявлены ошибки, требующие уведомления
    sendToSmtp mail {
      // Параметры smtp сервера (host, user, password)
 
      // Отправитель писем
      fromAddress = 'monitor@company.ru'
      // Адресаты писем
      toAddress = 'user1@company.ru,user2@company.ru'
    }
}
Пояснения

:

  • Для хранения статусов правил разрешается указывать любую таблицу БД, которая поддерживает добавление и изменение записей в Getl;
  • Запрос правила должен возвращать одну запись с полем value, в котором содержится последняя максимальная дата-время контролируемого поля таблицы;
  • Если в таблице контролируются группы записей, то запрос должен возвращать имена групп в поле code и значения максимальной даты-времени для каждой из групп;
  • Если есть возможность, что в результате сбоев в таблице могут находиться  данные с временем, которое больше текущего времени (в будущем), в запросах рекомендуется указывать фильтрацию записей, чтобы они не превышали текущую дату сервера;
     

    Вызов проверки статуса

    При вызове Check происходят следующие действия:
  • Проверяется существование таблицы, указанной в statusTable, если ее в БД не существует, создается новая с требуемой структурой
  • Анализируются записи текущего статуса из таблицы statusTable;
  • Выполняются запросы всех правил, у которых последний статус проверки просрочен по заданной частоте проверки и вычисляется корректность отставания данных с запросов от текущего времени сервера;
  • Правила, у которых впервые зарегистрировано отставание, помечаются к уведомлению;
  • Правила, у которых ранее уже было отставание и время регистрации ошибки превысило время частоты повтора рассылки уведомлений, помечаются к уведомлению;
  • Статус проверенных правил записывается в таблицу statusTable
    В случае, если одному из правил было выставлено требование уведомления (новая ошибка или истекшее время проверки существующей ошибки), то check вернет false, иначе будет возвращено true.
     

    Рассылка писем

После вызова проверки статуса правил check можно разослать письмами уведомления по обнаруженным ошибкам с помощью метода sendToSmtp, которому требуется указать объект Getl mail. Письма рассылаются в HTML формате, в виде таблицы с выявленными ошибками. Возможность управлять повторными уведомлениями позволяет более тонко расписать правила выявления ошибок и не спамить письмами при каждой проверке правила о ранее обнаруженной и действующей ошибки. В случае, если ошибка зарегистрирована как открытый инцидент в системе инцидентов компании и уведомления о ней временно требуется выключить, достаточно в таблице статусов statusTable проставить флаг поля open_incident в true. Правило будет продолжать отслеживать свой статус, но не будет выставлять уведомления, что произошла ошибка и являться инициатором рассылки писем.
