Работа с конфигурациями

 
Getl поддерживает работу с конфигурационными файлами в форматах Slurper (Groovy скрипты) и Encode Store (защищенные зашифрованные файлы). Файлы конфигурацию можно хранить в ресурсных файлах или на файловой системе. В файлах конфигурации можно разделять хранение по средам выполнения.
 
Напишем файл конфигурации connections.conf, в котором описаны подключения к Vertica и ftp для dev и prod сред выполнений:
environment {
  dev {
    connections {
      ver1 {
        connectHost = ‘localhost’
        connectDatabase = ‘vmart’
      }
    }
 
    files {
      ftp1 {
        server = ‘localhost’
      }
    }
  }
 
  prod {
    connections {
      ver1 {
        connectHost = ‘vertica.host1’
        connectDatabase = ‘vmart’
      }
    }
 
    files {
      ftp1 {
        server = ‘ftp.host1’
      }
    }
  }
}
 
Также напишем файл logins.conf, который будет хранить логины и пароли подключений к Vertica и ftp серверу без разделения по средам выполнения:
security {
  logins {
    ver1 {
      dbadmin = ‘admin’
      etluser = ‘etl’
      biuser = ‘bi’
    }
  }
  files {
    ftp1 {
      login = ‘ftpuser’
      password = ‘user’
    }
  }
}
 
Зашифруем этот файл и скопируем в ресурсный директорий проекта через проводник, указав, что значения действительный для любых сред выполнения. Для этого напишем небольшой скрипт, который считывает обычный конфигурационный файл и записывает его информацию в зашифрованный файл:
configuration {
  clear()
  load ‘<Путь к исходному файлу>/logins.conf’
  saveEncrypt configContent.security, ‘<Путь хранения файла>/logins.store’, ‘all’, ‘secret_password’
}
 
Пояснения:
За работу с конфигурациями в Getl отвечает блок configuration. В приведенном коде была очищена полностью текущая конфигурация приложения, загружен файл конфигурации и его секция security сохранена в защищенный от просмотра файл.
 
Теперь можно использовать эти конфигурации при работе с источниками:
configuration {
  // Выводим в лог имя текущей среды выполнения приложения
  logFine “Используется среда выполнения: $environment”
 
  // Загрузка файлов конфигурации
  load ‘connections.conf’
  loadEncrypt ‘logins.store’, ‘all’, ‘secret_password’
}
 
verticaConnection {
  // Будут использованы параметры подключения из connections.conf
  useConfig ‘ver1’
 
  // Будет использовано хранилище логинов из logins.store
  loginsConfigStory = ‘logins.ver1’
 
  // Установить пароль подключения для логина biuser
  useLogin biuser
 
  // Соединится с сервером
  connect()
}
 
ftp {
  // Использовать атрибуты подключения и логин с паролем с конфигурации
  useConfig ‘ftp1’
 
  // Соединится с сервером
  connect()
}
 
Пояснения:
  • Для JDBC соединений можно описать в отдельной секции доступные логины и их пароли для использования. При указании в атрибуте loginConfigStory имени секции конфигурации, из нее по именам логинов берутся пароли. Если такой секции нет или в ней отсутствует требуемый логин, будет сгенерирована ошибка.
  • При загрузке множества конфигураций Getl объединяет данные загружаемых секций и параметров. Таким образом после загрузки двух файлов конфигурации для ftp сервера в configContent контента конфигурации приложения в секции files.ftp будут присутствовать и параметры подключения к серверу и логин с паролем, хотя они находятся в разных файлах.
     
Для случаев, когда скрипт имеет параметры, но требуется их установка по умолчанию, в методе скрипта init можно указать загрузку конфигурации с дефолтными значениями:
@Field Integer count = 1
@Field String mask
 
void init() {
  configuration {
    load ‘resource:/default.parameters.conf’
    readFields ‘myscript1’
  }
}
 
void check() {
  assert (count?:0) > 0
  assert mask != null
}
 
При вызове данного скрипта, для параметра count будет сразу установлено значение 1. Далее будет вызван метод init, который считает конфигурационный файл и значения параметров заполнит из переменных из секции myscrip1.
 
Если при вызове скрипта были явно заданы значения параметров, то они будут установлены поверх. Таким образом можно организовать 3х уровневую передачу параметров, задавая значения по иерархии:
скрипт->конфигурационные файлы->параметры вызова
 
Создадим конфигурационный файл default.parameters.conf в ресурсном директории проекта:
myscript1 {
  mask = ‘mask1’
}
 
Пояснения:
  • Теперь при вызове скрипта без параметров у него параметр count будет равен 1, а параметр mask равен mask2.
  • При вызове скрипта из командной строки с параметром vars.count=2, в скрипте у параметра count станет значение 2.
  • При запуске скрипта через лаунчер Getl приложению по умолчанию задается среда выполнения “prod”.
     
 Для указания другой среды выполнения можно указать параметр environment:
java -cp "<jar скрипта>" getl.lang.Getl runclass=demo.MyScript environment=dev
 
P.S. В конфигурации можно указать любое количество сред выполнения, например ввести test и preprod.
